1. Schnellstart

Wenn das System neu aufgesetzt oder geprueft werden soll, ist dies die kuerzeste sinnvolle Reihenfolge.

  1. .env.local bzw. produktive Environment-Werte setzen.
  2. Lokale Verbindungsdatei auf Basis von config/amagno_connections.json anlegen.
  3. Zielpfade, FTP- oder SQL-Ziele pruefen.
  4. bin/console amagno:sync --all-connections --dry-run ausfuehren.
  5. Wenn Signaturpruefung aktiv ist: bin/console amagno:verify-signatures --all-connections --dry-run.
  6. Erst danach geplante Tasks aktivieren.
Faustregel: Erst mit --dry-run pruefen, dann produktive Rueckschreibungen und Stempel aktiv nutzen.

2. Betriebsrelevante Dateien

Datei Zweck Hinweis
.env Defaultwerte fuer Base-URI, Credentials, Matching-Datei, Checkpoint-Verzeichnis In Produktion besser ueber echte Env-Variablen oder .env.local ueberschreiben
config/services.yaml verbindet Parameter mit Symfony-Services relevant, wenn Dateipfade, Umgebungsvariablen oder zentrale Defaults wie amagno.stamp_retries geaendert werden
config/amagno_connections.json Beispiel bzw. aktive Struktur fuer Verbindungen enthaelt Credentials und mehrere Konfigurationen
oldProject/matching.json Mapping fuer den Export wird ueber die Settings-Oberflaeche gepflegt; nur diese Matching-Pflege ist aus dem Legacy-Ordner noch angebunden
amagno_sync.bat Beispiel fuer einen Windows-Task mit Netzlaufwerk-Handling hilfreich bei UNC-Shares und Service-Accounts

3. Einrichtung

Umgebungswerte

AMAGNO_BASE_URI=https://amagno.me
AMAGNO_API_USERNAME=
AMAGNO_API_PASSWORD=
AMAGNO_API_AUTH_TYPE=
AMAGNO_CREDENTIAL_ID=1
AMAGNO_CONNECTIONS_FILE=config/amagno_connections.local.json
AMAGNO_MATCHING_FILE=oldProject/matching.json
AMAGNO_TEMPLATE_DIR=oldProject
AMAGNO_CHECKPOINT_DIR=var/checkpoints

Verbindungsdatei

Die JSON-Datei besteht aus zwei Listen:

  • credentials: technische Amagno-Logins
  • configurations: konkrete fachliche Verbindungen mit Magnet, Vault, Exportziel oder einem reinen Signatur-Check

Beispiel fuer reinen Datenexport:

{
  "credentials": [
    {
      "cid": 1,
      "base_uri": "https://amagno.me",
      "username": "demo@example.com",
      "password": "SECRET",
      "auth_type": null
    }
  ],
  "configurations": [
    {
      "id": "nev-onprem-export",
      "active": true,
      "credential_id": 1,
      "vault_id": "GUID",
      "magnet_id": "GUID",
      "profile": "Nevaris Export",
      "template": "debitoren",
      "system": "onprem",
      "export": "local",
      "folder": "\\\\server\\share\\amagno"
    }
  ]
}

Beispiel fuer reinen Signature Check:

{
  "credentials": [
    {
      "cid": 1,
      "base_uri": "https://amagno.me",
      "username": "demo@example.com",
      "password": "SECRET",
      "auth_type": null
    }
  ],
  "configurations": [
    {
      "id": "nev-onprem-signature-check",
      "active": true,
      "credential_id": 1,
      "vault_id": "GUID",
      "magnet_id": "GUID",
      "signature_check": {
        "required_tag": "GUID-ZU-PRUEFEN-DURCH",
        "confirmed_tag": "GUID-GEPRUEFT-DURCH",
        "success_stamp": "GUID-ERFOLGREICH",
        "checkpoint_key": "nev-onprem-signature-check"
      }
    }
  ]
}
Wichtig: Wenn credential_id nicht zu einem Eintrag in credentials passt, startet der Lauf nicht. Fuer einen reinen Signature-Check werden keine Exportfelder benoetigt, vault_id muss aktuell aber trotzdem befuellt sein.

Settings-Oberflaeche

Die Weboberflaeche unter /settings bindet weiterhin oldProject/settings.php ein und speichert Matchings ueber /save.php bzw. /settings/save.php. Alte Export- oder API-Endpunkte aus dem Legacy-Projekt sind nicht mehr Teil des Symfony-Routings.

  • Aktive Webrouten: /, /settings, /save.php, /settings/save.php
  • Produktiver Export: bin/console amagno:sync
  • Produktive Signaturpruefung: bin/console amagno:verify-signatures

4. Regelbetrieb und Jobs

Sync manuell

bin/console amagno:sync --all-connections
bin/console amagno:sync --connection=nev-onprem
bin/console amagno:sync --all-connections --dry-run
bin/console amagno:sync --all-connections --use-checkpoint

Wichtige Optionen

  • --all-connections: alle aktiven Profile nacheinander
  • --connection=<id>: genau eine Verbindung
  • --dry-run: keine Exporte, keine Rueckschreibungen
  • --use-checkpoint: nur geaenderte Dokumente seit letztem Lauf
  • --since=ISO-DATUM: manueller Startpunkt

Windows-Aufgabenplanung

Fuer produktive Laeufe unter Windows ist ein Batch-Skript sinnvoll, besonders bei UNC-Freigaben. Ein vorhandenes Beispiel liegt in amagno_sync.bat.

@echo off
set APP_ENV=prod
set APP_DEBUG=0
cd /d C:\inetpub\amagno_nev_interface
php bin\console amagno:sync --all-connections --use-checkpoint >> C:\logs\amagno_sync.log 2>&1
Empfehlung: Den produktiven Sync zuerst ohne Signaturpruefung stabil betreiben und die zweite Task erst danach zuschalten.

5. Signaturpruefung

Die Signaturpruefung ist optional und wird pro Verbindung ueber den Block signature_check aktiviert.

"signature_check": {
  "required_tag": "GUID-ZU-PRUEFEN-DURCH",
  "confirmed_tag": "GUID-GEPRUEFT-DURCH",
  "result_attribute": "GUID-ERGEBNIS",
  "success_stamp": "GUID-ERFOLGREICH",
  "complete_stamp": "GUID-VOLLSTAENDIG",
  "incomplete_stamp": "GUID-UNVOLLSTAENDIG",
  "checkpoint_key": "nev-onprem-signature-check"
}

Manuelle Aufrufe

bin/console amagno:verify-signatures --all-connections --dry-run
bin/console amagno:verify-signatures --connection=nev-onprem --use-checkpoint
  • required_tag: erwartete Pruefer
  • confirmed_tag: vorhandene Unterschriften bzw. Freigaben
  • result_attribute: Textfeld fuer das Pruefergebnis
  • success_stamp bzw. complete_stamp: Stempel fuer vollstaendige Dokumente
  • incomplete_stamp: Stempel fuer unvollstaendige Dokumente
Wichtig: Doppelte Namen in required_tag oder confirmed_tag werden absichtlich ignoriert. Die Pruefung erwartet nur, dass jeder erforderliche Name mindestens einmal in den bestaetigten Namen wiedergefunden wird.

6. Checkpoints

Checkpoints verhindern, dass bei jedem Lauf alle Dokumente erneut bearbeitet werden. Gespeichert wird der letzte bekannte changeDate-Wert als JSON-Datei im konfigurierten Checkpoint-Verzeichnis.

  • Standardpfad laut Env: var/checkpoints
  • Der Dateiname wird aus dem Checkpoint-Key erzeugt.
  • Ein geloeschter Checkpoint fuehrt dazu, dass beim naechsten Lauf wieder breiter gelesen wird.
Wichtig: Wenn "keine neuen Dokumente" gemeldet wird, obwohl etwas erwartet wird, ist der Checkpoint einer der ersten Pruefpunkte.

7. Logs und Kontrolle

Was beobachten?

  • Exit-Code des Batch- oder Console-Laufs
  • eigene Task-Logs, z. B. in C:\logs oder var\log
  • Anzahl gefundener und verarbeiteter Dokumente
  • Rueckschreibungen von Fehlertexten, Erfolgs- oder Fehlerstempeln

Nutzliche Befehle

bin/console amagno:sync --all-connections --dry-run -vvv
bin/console amagno:verify-signatures --all-connections --dry-run -vvv

Die Anwendung verwendet Monolog. Je nach Umgebung landen Logs typischerweise in var/log/dev.log oder var/log/prod.log.

8. Typische Fehlerbilder

Symptom Ursache Pruefung
Credential-ID ... nicht vorhanden Verbindung verweist auf ungueltige Credential-ID credential_id gegen credentials[].cid pruefen
Keine Amagno Base URI weder Env noch Verbindung liefert base_uri .env.local und JSON vergleichen
Keine neuen Dokumente gefunden Checkpoint oder falscher Magnet mit --dry-run und ggf. ohne --use-checkpoint testen
Lokaler Export scheitert Zielordner existiert nicht oder Rechte fehlen Ordner manuell pruefen, ggf. Service-Account testen
Windows-Task kommt nicht auf Netzlaufwerk Task laeuft mit anderem Benutzerkontext Batch mit net use und expliziten Credentials nutzen
No route found for POST .../save.php Legacy-Settings-Route nicht verfuegbar SettingsActionController und Webrouting pruefen
404 auf /api/fibu_get, /api/fibu_deleteProfile oder oldProject/export.php alter Legacy-Endpunkt wird aufgerufen nicht mehr verwenden; Matchings ueber /settings speichern und Exporte per CLI starten

9. Regelmaessige Aufgaben

Vor einem Release

  • lokale Konfigurationsdatei sichern
  • --dry-run fuer Sync und Signaturpruefung ausfuehren
  • Zielpfade und Freigaben erreichbar pruefen

Im Stoerungsfall

  • zuletzt geaenderte Konfiguration identifizieren
  • mit einer einzelnen Verbindung testen
  • Checkpoints kontrollieren
  • erst danach globale Tasks erneut freigeben

Nicht vergessen

  • Die Settings-Oberflaeche unter /settings basiert auf Legacy-Code und ist nur fuer die Matching-Pflege angebunden.
  • Credentials und produktive JSON-Dateien nicht versehentlich ins Repository einchecken.
  • Bei grossen Aenderungen immer zuerst einen manuellen Lauf mit einer einzelnen Verbindung pruefen.
